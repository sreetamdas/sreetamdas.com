# Our fix: Single Source of Truth

Backend defines:

- Expected shape
- Constraints
- Conditional logic
- even the error copy

<Notes>

Core idea: make the rule data travel across the server-client interface, instead of re-implementing the rule in
React/on your FE, and then again in Elixir/on your BE.

</Notes>

# JSON Schema

- Standard way to describe data shape
- `type`, `properties`, `required`
- Conditional logic with `allOf`, `if`, `then`

<Notes>

What we use to make those rules travel across the server-client barrier

- `type: "object"` means the payload is an object
- `properties` lists the fields
- `required` lists mandatory fields
- `allOf` / `if` / `then` can model conditional constraints

JSON Schema can say what data is valid, but it does not really tell us how
to render a great product form. That is where JSF adds a thin layer of UI-oriented metadata.

</Notes>

---

# JSON Schema Form

- Headless
- Schema in
- Fields out
- Validation out

<Notes>
does not render inputs. give it a schema and it returns a list of fields, plus validation fn.

not "yet another React component library."

</Notes>

---

# Core API

```tsx
const { fields, handleValidation } = createHeadlessForm(schema, {
  initialValues,
});
```

- No components
- No CSS
- No design system
- Bring your own form library

<Notes>

`createHeadlessForm()` is the main entrypoint. It returns fields that your UI can render, and a
`handleValidation()` function that validates values against the schema.

fields rendered with our internal form system: design system Norma and Formik.

Different domain teams can still have custom-looking/themed form elements, while JSF takes care of the schema and validation

</Notes>

# Architecture Overview

- Backend defines the schemas
- JSON Schema Form transforms schema → fields
- FE handles rendering

<Notes>

Backend authors the schema based on requested form - can change depending on country, employment type, role, etc.

each layer has one job, and the contract between them is just data.

</Notes>

# Conditional fields

- `if`
- `then`
- `else`
- `allOf`

```json
{
  "type": "object",
  "properties": {
    "employment_type": {
      "type": "string",
      "enum": ["employee", "contractor"]
    },
    "annual_salary": {
      "type": "number",
      "minimum": 0
    },
    "hourly_rate": {
      "type": "number",
      "minimum": 0
    }
  },
  "required": ["employment_type"],
  "allOf": [
    {
      "if": {
        "properties": { "employment_type": { "const": "employee" } },
        "required": ["employment_type"]
      },
      "then": {
        "required": ["annual_salary"],
        "not": { "required": ["hourly_rate"] }
      }
    },
    {
      "if": {
        "properties": {
          "employment_type": { "const": "contractor" }
        },
        "required": ["employment_type"]
      },
      "then": {
        "required": ["hourly_rate"],
        "not": { "required": ["annual_salary"] }
      }
    }
  ]
}
```

# JSON Logic

- Cross-field rules
- Computed values
- Date math
- Custom operators

<Notes>
JSON Schema conditionals cover a lot, but richer business logic often wants intra-form, cross-field dependencies.

This is where `x-jsf-logic` comes in. Remote uses it for validations and computed values. Dragon and
server-side validation can both evaluate those rules.

Good codebase-backed examples to pull into a later pass:

- date difference in months from the lambda test
- `date_add_days` / date helper operators from the kit
- country-specific salary/probation computed values from Tiger RMT components

</Notes>

# So, why JSF?

- Backend as author
- UI as consumer
- Validation everywhere

<Notes>
This is the closing distillation.

The point is not "everyone should use our exact library." The point is the architecture:

When form rules change by country, product, employment type, or legal requirement, the rule should not
live inside a React component. Put it in data, ship it across the wire, render it headlessly, and
validate it wherever the data enters the system.

</Notes>
