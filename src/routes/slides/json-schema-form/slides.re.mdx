# JSON Schema

- Standard way to describe data shape
- `type`, `properties`, `required`
- Conditional logic with `allOf`

<Notes>
Keep this short. The audience does not need a full JSON Schema tutorial.

Show enough that the rest of the talk is legible:

- `type: "object"` means the payload is an object
- `properties` lists the fields
- `required` lists mandatory fields
- `allOf` / `if` / `then` can model conditional constraints

Then make the transition: JSON Schema can say what data is valid, but it does not really tell us how
to render a great product form. That is where JSF adds a thin layer of UI-oriented metadata.

</Notes>

---

# JSON Schema Form

<Steps>

- Headless
- Schema in
- Fields out
- Validation out

</Steps>

<Notes>
This is the simplest definition of the library.

`@remoteoss/json-schema-form` is headless. It does not render inputs. You give it a schema and it
gives you a list of fields, plus validation.

The important thing for the audience: this is not "yet another React component library." It is closer
to a compiler step from schema to field model.

</Notes>

---

# Core API

```tsx
const { fields, handleValidation } = createHeadlessForm(schema, {
  initialValues,
});
```

<Notes>

`createHeadlessForm()` is the main entrypoint. It returns fields that your UI can render, and a
`handleValidation()` function that validates values against the schema.

There is also `modify()`, which lets consumers transform a schema without mutating the backend source
schema directly.

</Notes>

# `x-jsf-*`

- `x-jsf-presentation`
- `x-jsf-order`
- `x-jsf-errorMessage`
- `x-jsf-logic`

<Notes>
JSON Schema lets consumers define custom keywords. JSF uses `x-jsf-*` keys for form-specific
metadata.

The backend can ship these keys in the same schema payload. Servers can ignore UI hints that do not
matter to backend validation; the frontend can use them for rendering.

Important codebase detail: Dragon's API schema transform explicitly preserves `x-jsf-*` keys because
JSF expects those exact names. It camel-cases field names where needed, but not the extension keys.

</Notes>

# Presentation

- Input type
- Description
- Options
- Field order

<Notes>
This is the "how does JSON Schema become a form?" slide.

Basic JSON Schema tells us "string", "number", "required", "minimum". Presentation metadata tells
Dragon whether that should be a text field, money input, select, radio group, file upload, statement,
fieldset, etc.

Backend helper example from Tiger: `Remote.RmtJson.Components` has builders like `text_field`,
`number_field`, `money_field`, `radio_field`, `select_field`, and helpers like `add_property`,
`add_required`, `add_all_of`, and `add_fieldset`.

</Notes>

# JSON Logic

- Cross-field rules
- Computed values
- Date math
- Custom operators

<Notes>
JSON Schema conditionals cover a lot, but richer business logic often wants cross-field calculations.

This is where `x-jsf-logic` comes in. Remote uses it for validations and computed values. Dragon and
server-side validation can both evaluate those rules.

Good codebase-backed examples to pull into a later pass:

# Lessons

- Rules as data
- UI as consumer
- Backend as author
- Validation everywhere

<Notes>
This is the closing distillation.

The point is not "everyone should use our exact library." The point is the architecture:

When form rules change by country, product, employment type, or legal requirement, the rule should not
live inside a React component. Put it in data, ship it across the wire, render it headlessly, and
validate it wherever the data enters the system.

</Notes>
