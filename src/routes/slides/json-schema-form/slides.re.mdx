# JSON Schema

- Standard way to describe data shape
- `type`, `properties`, `required`
- Conditional logic with `allOf`

<Notes>
Keep this short. The audience does not need a full JSON Schema tutorial.

Show enough that the rest of the talk is legible:

- `type: "object"` means the payload is an object
- `properties` lists the fields
- `required` lists mandatory fields
- `allOf` / `if` / `then` can model conditional constraints

Then make the transition: JSON Schema can say what data is valid, but it does not really tell us how
to render a great product form. That is where JSF adds a thin layer of UI-oriented metadata.

</Notes>

# JSON Schema Form

<Steps>

- Headless
- Schema in
- Fields out
- Validation out

</Steps>

<Notes>
This is the simplest definition of the library.

`@remoteoss/json-schema-form` is headless. It does not render inputs. You give it a schema and it
gives you a list of fields, plus validation.

The important thing for the audience: this is not "yet another React component library." It is closer
to a compiler step from schema to field model.

</Notes>

---

# Core API

```tsx
const { fields, handleValidation } = createHeadlessForm(schema, {
  initialValues,
});
```

<Notes>

`createHeadlessForm()` is the main entrypoint. It returns fields that your UI can render, and a
`handleValidation()` function that validates values against the schema.

There is also `modify()`, which lets consumers transform a schema without mutating the backend source
schema directly.

</Notes>

# `x-jsf-*`

- `x-jsf-presentation`
- `x-jsf-order`
- `x-jsf-errorMessage`
- `x-jsf-logic`

<Notes>
JSON Schema lets consumers define custom keywords. JSF uses `x-jsf-*` keys for form-specific
metadata.

The backend can ship these keys in the same schema payload. Servers can ignore UI hints that do not
matter to backend validation; the frontend can use them for rendering.

Important codebase detail: Dragon's API schema transform explicitly preserves `x-jsf-*` keys because
JSF expects those exact names. It camel-cases field names where needed, but not the extension keys.

</Notes>

# JSON Logic examples

```json
// Simple: 1 == 1
{ "==": [1, 1] }
// → true
```

```json
// Compound: (3 > 1) && (1 < 3)
{ "and": [{ ">": [3, 1] }, { "<": [1, 3] }] }
// → true
```

```json
// Data-driven: read from input
{ "var": "a" }
// with data { "a": 1, "b": 2 }
// → 1
```

```json
// Complex: pie is ready?
{
  "and": [
    { "<": [{ "var": "temp" }, 110] },
    { "==": [{ "var": "pie.filling" }, "apple"] }
  ]
}
// with { "temp": 100, "pie": { "filling": "apple" } }
// → true
```

<Notes>
These are the canonical JSON Logic examples from jsonlogic.com.

The format is always: `{"operator": [values]}`. Operator is always the key. Values are an array.
Each value can be a string, number, boolean, array, or null.

The `var` operator reads from the data object — this is how rules become data-driven instead of
hardcoded.

The complex example shows how you nest rules: the `<` comparison uses `{ "var": "temp" }` as its
first operand, meaning "read the temp field from the data and compare it to 110."

This is exactly the pattern Remote uses for cross-field validation: salary > minimum for country,
or probation period depends on employment type. The rule is just JSON — no code to deploy, no
functions to write.

</Notes>

# Server validation

- Tiger calls validator
- Same schema
- Same JSF logic
- Same error shape

<Notes>
This is probably the biggest "wow" moment for an Elixir audience.

The old path: Tiger can call an AWS Lambda client for JSON Logic validation.

The newer path: Tiger can route through `ExJsfLogic`, a local Port-based sidecar that runs
`@remoteoss/remote-json-schema-form-kit`.

Dragon also has a `json-schemas-validate` lambda where the core code is essentially:

`createHeadlessForm(schema).handleValidation(data)`

That is the concrete proof of the single-source-of-truth story.

</Notes>

# Real gotchas

- snake_case / camelCase
- Nested conditionals
- Schema mutation
- Partial forms

<Notes>
These are good because they make the talk credible.

Dragon's wrapper exists because the core idea is clean, but product reality is not.

Specific gotchas from code:

- API schemas arrive with snake_case field names, while Dragon often works in camelCase
- `x-jsf-*` extension keys must be preserved exactly
- JSF has had mutation issues around `initialValues`, so Dragon defensively deep-copies
- nested conditional fieldsets may need multiple validation passes
- partial forms need picked/omitted fields so hidden steps do not block validation

</Notes>

# Multi-pass validation

- One pass
- One visibility level
- Repeat until stable

<Notes>
This is a nice deep-dive slide if time allows.

fields. The code computes the maximum nesting depth and short-circuits when values stop changing.

This is a good lesson slide: "Headless" does not mean "no integration work." It means the integration
work has a clean boundary.

# Lessons

- Rules as data
- UI as consumer
- Backend as author
- Validation everywhere

<Notes>
This is the closing distillation.

The point is not "everyone should use our exact library." The point is the architecture:

When form rules change by country, product, employment type, or legal requirement, the rule should not
live inside a React component. Put it in data, ship it across the wire, render it headlessly, and
validate it wherever the data enters the system.

</Notes>
